Chapter snapshot

Master the IPOS story (input → processing → output → storage), bits vs bytes, binary / decimal / hex / octal, capacity vs throughput units, CPU clock, and a disciplined troubleshooting sequence. Exam items often look like “which phase?” or “which unit is larger?”

Think of this chapter as the vocabulary of how computers talk about themselves. If you know whether a number measures size or speed, and whether a symptom needs a logic fix or a process fix, harder objectives later (networking, security) become easier to place in context.

Computing flow: input, processing, output, storage (IPOS)

Computers look complicated, but almost every task you perform—typing an email, joining a video call, printing a receipt—follows the same story. Data is captured, transformed, shown or sent somewhere, and often kept for later. CompTIA labels that story IPOS. Exam writers love short scenarios because they check whether you understand roles, not memorized brand names.

Think of input as anything that turns a real-world fact into digital signals: keystrokes, touchscreen coordinates, microphone audio, camera frames, barcode scans, or packets arriving from the network. Nothing meaningful happens until those signals exist inside the machine. Processing is where meaning is applied—the CPU runs program instructions, the GPU handles massive parallel work like drawing 3D scenes, and mobile SoCs combine CPU, GPU, and radios on one chip. Output turns results back into something humans or other systems can use: pixels on a display, sound from speakers, paper from a printer, or packets sent to another device. Storage answers “what happens when the power cycles or the user comes back tomorrow?” RAM holds what you are working on right now but usually empties when power drops; disks, SSDs, and flash keep bits without power; optical discs still appear on exams as a distribution and archive medium.

Walkthrough: You dictate a note into your phone. The microphone and ADC produce digital audio (input). The speech-to-text app runs on the processor (processing). You read the transcribed text on screen (output). When you tap Save, the note file lands in flash storage (storage).
  • Input — Keyboard, mouse, touch, mic, camera, scanner, NFC, NIC receiving data.
  • Processing — CPU instructions, GPU rendering, SoC bundles on phones.
  • Output — Display, speakers, printer, haptics, network transmit.
  • Storage — RAM (volatile workspace), SSD/HDD/NVMe, optical, removable flash.
Trick questions: “The GPU lights up pixels” is still processing until those pixels hit the panel; showing the image is output. Network storage (cloud sync) still ends in a stored copy somewhere—even if you do not see the drive.

Bits, bytes, and number systems

Inside the machine, everything is ultimately binary: voltage high or low, gate on or off. A bit is that smallest fact. Group eight bits and you get a byte, which can represent one of 256 patterns—enough for a small integer, a keyboard character in many encodings, or part of a color. We use decimal in daily life because ten fingers; we use hex and octal in IT because they map cleanly to binary (one hex digit = four bits; one octal digit = three bits), so long strings of bits become shorter to read and type.

When you see a MAC address like A4:83:E7:12:34:56, each pair is one byte in hex. When a web color is #2A5F9C, each pair is one byte of red, green, blue. You do not have to convert huge numbers by hand on Tech+, but you do need to know what each base is for and that binary place values in a byte run 128, 64, 32, 16, 8, 4, 2, 1.

A nibble (half a byte, four bits) shows up rarely—know the word in case a question mentions it.

SystemBaseDigits / symbolsTypical use
Binary20–1Machine language, masks, addressing
Decimal100–9Human-friendly numbers
Hexadecimal160–9, A–FMAC addresses, memory dumps, HTML color #RRGGBB

For binary place values in one byte (right to left), positions are 1, 2, 4, 8, 16, 32, 64, 128. Summing “on” bits gives the decimal value (e.g. 1010 0000 in one byte = 128 + 32 = 160).

Hex pairs = one byte: Two hex digits represent 0–255. Example: #FF = 255 decimal. In a color like #336699, each pair is one byte of red, green, blue.

Storage capacity units (KB → PB)

Capacity answers “how much can we hold?”—different from “how fast can we move data?” (throughput, next section). Engineers often use powers of 1024 because memory and addressing are naturally binary: each step up multiplies by 210. Textbooks and some exam items simplify that to 1,000 steps between KB, MB, GB (SI-style). Both framings appear in the wild. What you must never miss is the ordering: KB < MB < GB < TB < PB, and that a terabyte-class drive holds vastly more than a gigabyte-class USB stick.

UnitApproximate scaleIntuition
KB1024 bytes (often rounded to 1,000 in prose)Small text file, short email
MB1024 KBPhotos, documents, small apps
GB1024 MBOS installs, games, video
TB1024 GBWorkstations, NAS, backups
PB1024 TBData centers, cloud-scale
Why your “1 TB” drive looks smaller in Windows: Manufacturers often advertise decimal terabytes (1012 bytes). The OS may report in binary-ish units, so the number looks like less than 1 TB even though the hardware is the same. For the exam, prioritize “which unit is larger?” over decimal precision.

Throughput: bps, Mbps, Gbps (and bytes per second)

Networks sell speed in bits per second because the physical layer moves bits; ISPs and Wi-Fi specs say Mbps or Gbps. File-copy dialogs often show MB/s because humans think in file size (bytes). The bridge between the two is always divide by 8 when converting from bits per second to bytes per second (and multiply by 8 going the other way).

Worked idea: A 1 Gbps (1000 Mbps) Ethernet link can move at most about 125 MB/s of payload in math-class theory. Real life is lower because of protocol overhead, retransmits, and disk limits—but the exam repeatedly checks whether you remembered the 8 bits = 1 byte step.

Watch capitalization: Mb or Mbps = megabits; MB = megabytes. Mixing them up is one of the most common wrong answers.

CPU speed: MHz, GHz, and what else matters

The clock is a metronome: it does not do work by itself, but it bounds how fast the CPU can step through its internal pipeline. MHz is millions of ticks per second; GHz is billions. A 3.6 GHz CPU ticks faster than a 2.0 GHz part of the same generation and core layout, but you cannot compare raw GHz across ten years of architecture change—a modern mid-range chip may crush an old “fast” GHz monster because of better branch prediction, cache, and instructions per cycle.

For Tech+, know the vocabulary: multicore (several CPUs on one package), multithreading (often two hardware threads per core), and cache (tiny, fast memory next to the core—L1 fastest/smallest, L3 larger/slower). Thermal throttling means the chip slows down if cooling cannot remove heat—another reason a laptop “feels slow” under load.

Troubleshooting methodology (learn the order)

Random fixes waste time and can make things worse (wrong driver, wrong setting, lost data). Professional troubleshooting is evidence-driven: you gather clues, form a guess, test that guess in the smallest way possible, then fix and prove the system is healthy again.

Imagine a user says “the Internet is broken.” That is not one problem—it is a symptom. You confirm whether one site fails or all sites fail, whether DNS names fail but numeric IPs work, whether only Wi-Fi is down while Ethernet works. Each answer narrows the fault domain (browser vs account vs Wi-Fi vs ISP). That narrowing is what “identify and isolate” means in practice.

  1. Identify the problem — Question the user; reproduce; note error messages, beeps, LEDs, recent changes; separate one problem from unrelated noise.
  2. Form a theory of probable cause — Rank likely causes (software vs hardware vs network vs account).
  3. Test the theory — One change at a time; isolate variables (swap cable, different port, safe mode, ping gateway). If the theory is wrong, undo and try the next theory.
  4. Fix the issue — Implement the solution (patch, driver, setting, replacement part) based on what you proved.
  5. Verify full functionality and document — Confirm the symptom is gone and nothing else broke; record symptoms, cause, fix, and any follow-up in the ticket or knowledge base.
Exam angle: “Try random fixes until it works” is wrong. “Test one theory at a time, then verify” is the professional answer.

Motherboard, firmware, POST, and boot

The motherboard is the backplane that ties everything together: CPU socket, RAM slots, chipset, firmware chip, expansion slots (PCIe), headers for front-panel buttons, fan connectors, SATA/NVMe/M.2 for storage, and sometimes onboard video or sound. Traces carry high-speed signals—flexing a board, stripped screws, or liquid kills machines faster than “software problems.”

BIOS (older Basic Input/Output System) and UEFI (modern firmware interface) both run before the operating system. They initialize hardware, store baseline settings (boot order, date/time, security options), and hand off to a bootloader on disk. UEFI supports larger disks, faster boot, Secure Boot, and a mouse-friendly setup utility—but exam questions still say “BIOS” as a generic term for firmware setup.

POST (Power-On Self-Test) is the firmware’s sanity check: RAM presence, CPU, sometimes GPU. Failure often produces beep codes (if a speaker is present) or diagnostic LEDs on consumer boards. Random restarts or “no POST” point to power, RAM seating, overheating, or a dead component before you blame Windows.

A CMOS battery (coin cell) keeps the real-time clock and BIOS settings when AC power is off. If the machine “forgets” time or resets boot order, suspect a weak CMOS battery. Jumpers on the board can clear CMOS when a bad overclock or password lock bricks setup—power off first, follow the manual diagram.

Expansion, RAM form factors, and power basics

PCIe slots accept GPUs, NVMe adapters, extra NICs, and capture cards. Lanes (x1, x4, x16) describe electrical width—more lanes for more bandwidth (GPUs want x16). RAM for desktops often arrives as DIMM modules; laptops use smaller SODIMM. Match DDR generation (DDR4 vs DDR5) to the board; mixing speeds usually down-clocks to the slowest stick.

The PSU label lists wattage and 12V rail capacity—GPUs care about 12V delivery. Connectors include 24-pin ATX for the motherboard, CPU 4/8-pin near the socket, and PCIe power for high-end video cards. Modular PSUs let you detach unused cables for airflow. Never mix modular cables between vendors—pinouts differ and can fry hardware.

Binary drill (one byte): Bits 1100 1010 = 128 + 64 + 8 + 2 = 202 decimal. As hex: CA (12 = C, 10 = A). Practice converting your Wi-Fi channel or RAM size into “what does this number mean?” stories, not decimal long division under exam pressure.

Workstation setup (what “good” looks like)

Setting up a workstation is more than plugging in cables—it is making a system stable, updatable, and safe for daily use. CompTIA expects you to think like someone who could deploy a desk at a small office or help a family member at home.

Start with power quality: a surge strip is not the same as a UPS. A UPS gives you minutes to save work when utility power drops and often conditions noisy power. Route cables so nothing kinks, trips people, or blocks vents. Install Windows (or join a domain), create a standard user for everyday tasks, and reserve administrator elevation for installs and fixes—malware loves an always-admin account.

Confirm Device Manager is clean (no yellow bangs), Ethernet or Wi-Fi gets an address, and Windows Update has run at least once. Attach monitor, keyboard, mouse, and any dock or USB hub; verify sound and webcam in Settings before you call the job done.

Devices & internal components

Form factors exist because different jobs need different balances of power, portability, cost, and upgrade path. A tower desktop has room for full-size GPUs and multiple drives; a laptop trades expansion for mobility; a thin client assumes the “real computer” lives in the data center; a server prioritizes uptime, remote management, and redundant parts.

Form factorNotes for the exam
Desktop / tower / SFFModular upgrades (GPU, RAM, storage); separate monitor
Laptop / 2-in-1Integrated display; battery; mobile NICs; often soldered RAM/SSD on ultrabooks
Tablet / smartphoneTouch-first; ARM SoCs common; cellular + Wi-Fi

Thin clientMinimal local OS; relies on remote session / VDI / cloud app
IoT / embeddedSensors, limited UI, firmware, long-lived installs

Inside the case, the motherboard is the nervous system: traces and chipsets route signals between CPU, RAM, expansion slots, storage buses, and onboard controllers. The CPU fetches and executes instructions; it is dumb without software, but nothing runs without it. RAM is the scratchpad—large enough to hold running apps and OS cache, and orders of magnitude faster than disk, which is why “add RAM” fixes some slowdowns (until you saturate something else). The GPU is optional on office PCs but essential for 3D, video editing, and modern composited UIs. The PSU must supply stable 12V/5V/3.3V rails sized for peak GPU+CPU load; under-spec PSUs cause mysterious reboots. Storage and the NIC (wired or Wi-Fi module) sit on buses (SATA, NVMe, PCIe, M.2). Cooling moves heat from dies to air; if airflow is blocked, the system throttles or shuts down to protect silicon.

Storage types & memory behavior

HDDs store bits as magnetic patterns on spinning platters. A moving head seeks tracks—fast sequential reads, slower random small reads, vulnerable to shock. SSDs store charge in flash cells; no seek time, silent, great random I/O, but wear and cost per GB still matter at huge scale. NVMe puts flash on a PCIe lane with a protocol designed for SSDs (not the older SCSI mindset of SATA), so latency drops and queue depth scales—why NVMe “feels instant” for boot and app launch.

Volatile vs non-volatile is the power question: pull the plug on RAM and its contents are gone; SSD/HDD/ROM retain data. Local means you can point at the device; cloud means someone else’s data center, though sync tools keep local copies that blur the line for users.

Decision thinking: Bulk cold archive → cheap HDD. Everyday laptop → SSD/NVMe. Kiosk or embedded → small flash or eMMC. Exam questions often ask “which is fastest?” or “which is volatile?”—not “which brand?”

Peripherals, printers, and drivers

Peripherals extend what the core PC can sense or act on. When you plug something in, the OS asks “what are you?” using identifiers exposed over USB or Bluetooth. Plug and Play automates IRQ/DMA/memory assignment compared to the bad old days, but the OS still needs a driver—a translator between generic OS calls and vendor-specific hardware.

Inkjets spray liquid ink; great for photos and mixed color, cartridges can dry out. Laser printers fuse powdered toner with heat; excellent for sharp text and office volume. Thermal printers use heat on special paper—receipts, shipping labels. Network printers get an IP; shared USB printers rely on a host PC or print server. If print jobs stall, the queue, driver, and connectivity layer are your three suspects.

Ethernet cabling: Cat5e handles 1 Gbps in typical office lengths. Cat6/6a improves crosstalk performance and supports 10 Gbps at shorter distances. Wireless: Bluetooth, NFC, Wi-Fi

Bluetooth is a personal-area radio: low power, pairing, good for keyboards, audio, and short file swaps—not a replacement for building-wide LAN throughput. NFC is centimeters: tap-to-pay, tap-to-pair. Wi-Fi is your wireless LAN workhorse; it shares unlicensed spectrum with neighbors, microwaves, and Bluetooth, which is why placement, band choice, and channel width matter (Security chapter covers encryption).

Networking basics

Networks are layered. At the physical layer you have cables and radios; above that, devices speak Ethernet frames addressed by MAC. At layer 3, IP addresses let routers move packets across different subnets and across the world. Two machines on the same subnet talk locally (often via switch); to reach another network, traffic goes to a default gateway—usually your router’s LAN address.

Ports (like 443) are not jacks on the wall; they are numbers in TCP/UDP headers telling the OS which application should receive the segment. That is how one public IP can host web, mail, and VPN—each service listens on its own port.

LAN — Single site or logical grouping (home, office floor). WAN — Connects distant sites; the Internet is a public WAN. Router — Routes between subnets/networks; often includes NAT for home. Switch — Forwards frames inside a LAN by MAC. Modem — Terminates ISP link (cable/fiber/DSL). WAP — Adds Wi-Fi access to the wired LAN.

DHCP saves you from typing IP, mask, gateway, and DNS by hand—essential when laptops roam. DNS is the phone book: humans remember example.com; stacks need numbers. If DNS breaks, “the Internet looks down” even when raw IP still works—a classic isolate-the-layer lesson.

Client-server centralizes services (files, authentication, databases). Peer-to-peer treats nodes as equals—simple shares, some updates, but harder to secure and audit at scale.

Mini scenario: PC gets 169.254.x.x APIPA address → likely no DHCP reachability (bad Wi-Fi, rogue switch, DHCP server down). PC gets correct IP but cannot browse → try ping gateway, then ping 8.8.8.8, then DNS—each step tells you which layer failed.

Internet access types & Wi-Fi standards

  • Fiber — Glass, very high bandwidth, low latency; great for symmetric speeds.
  • Cable — Shared neighborhood segment; fast downloads; upload varies.
  • DSL — Copper phone lines; distance-sensitive.
  • Cellular (LTE/5G) — Mobile broadband; hotspot tethering.

Each Wi-Fi generation improved encoding, channel width, and multi-user tricks. 802.11n brought better 2.4/5 GHz speeds; 802.11ac pushed 5 GHz wave-2 gear; 802.11ax (Wi-Fi 6) targets dense environments; Wi-Fi 6E adds 6 GHz spectrum where regulators allow—more clean air, but shorter reach through walls. 2.4 GHz is crowded (Bluetooth, microwaves, old gear) but travels farther; 5 GHz offers more throughput with less range; 6 GHz is premium spectrum for new laptops and phones.

Fiber vs cable vs DSL vs cellular is an economics and physics story: fiber offers huge symmetrical bandwidth and low latency when deployed; cable shares neighborhood coax; DSL degrades with copper length; cellular roams but faces towers and data caps.

Cloud & virtualization

Virtualization decouples operating systems from bare metal. A hypervisor schedules CPU time, carves RAM, and virtualizes disks and NICs so each guest OS believes it owns hardware. That enables snapshots, live migration in data centers, and “rent a VM by the hour.”

ModelYou manageVendor manages
IaaSOS, apps, data, some networkingServers, storage, hypervisor, data center
PaaSApp and dataRuntime, middleware, scaling, patching platform
SaaSYour content/configEntire application stack

On-premises — You own the gear in your building. Public cloud — Shared provider (AWS/Azure/GCP style). Hybrid — Mix (e.g. AD on-prem + SaaS email). Hypervisor — Software/firmware that runs virtual machines: Type 1 (bare-metal, ESXi/Hyper-V on server) vs Type 2 (hosted on top of an OS, e.g. desktop VM apps).

IaaS is “give me VMs and disks”; you still patch the OS. PaaS is “give me a runtime and database service”; you focus on code. SaaS is “give me Gmail/Teams/365”—you configure, you do not compile the stack. The exam wants you to place a scenario into the right bucket: who patches the database engine? Who owns physical servers?

Basic maintenance & support mindset

  • Preventive — Dust filters, cable strain relief, surge/UPS, scheduled backups, patch Tuesday cadence.
  • ESD — Use anti-static strap/mat when handling boards; discharge before touching internals.
  • Environmental — Heat kills; ensure intake/exhaust clear; laptops on hard surfaces.

RAID concepts (recognition level)

RAID combines multiple disks for speed, redundancy, or both—usually via a hardware controller or OS software RAID. Tech+ expects vocabulary, not calculator math.

LevelIdeaTrade-off
0 (striping)Data split across disks—fast reads/writesNo redundancy; one disk dies → data loss
1 (mirroring)Two disks hold the same dataSurvives one disk failure; costs 50% capacity
5Striping + distributed paritySurvives one disk failure; needs at least three disks

Exam distractors may offer “RAID 0 is safest”—false. “RAID 1 is for redundancy”—true. NAS appliances and servers advertise these modes in their GUIs.

IPv4 essentials for workstations

You do not subnet for Tech+, but you must recognize normal vs broken addresses. Private ranges (not routed on the public Internet): 10.0.0.0/8, 172.16.0.0–172.31.255.255, 192.168.0.0/16. Home routers usually hand out 192.168.x.x. APIPA 169.254.0.0/16 means “I could not get DHCP”—check cable, Wi-Fi, or DHCP server. Loopback 127.0.0.1 tests the local TCP/IP stack.

A subnet mask (often 255.255.255.0 on small LANs) tells the host which bits identify the network vs the host. The default gateway is the router’s LAN IP on your subnet—traffic to other subnets goes there.

Displays, resolution, and projectors

Native resolution is the physical pixel grid of a panel—running “wrong” resolution looks soft or stretched. Aspect ratio (16:9 vs 16:10) matters for black bars. Refresh rate (Hz) is how often the image updates; higher feels smoother for motion. Mirroring shows the same image on laptop + projector; extended desktop gives a second workspace. If a conference room shows “no signal,” verify input source on the display, cable type (HDMI vs DP), and whether Win+P (or OS equivalent) is set to duplicate/extend.

Satellite, KVM, and docking

Satellite Internet reaches rural areas with high latency (signal to orbit and back)—fine for email and streaming with buffer, poor for competitive online gaming. KVM (Keyboard, Video, Mouse) switches let one console control multiple PCs—common in server closets. A docking station or Thunderbolt hub expands a laptop to many monitors, Ethernet, and USB with one cable.

Operating systems — purpose and types

An operating system is not “just another app.” It is the trusted layer between hardware and programs: it decides which process runs on which core, how RAM is shared, who may read a file, and how bytes move to disk or across the NIC. Without an OS, every program would need its own drivers for every printer and GPU—impossible to maintain.

The kernel stays resident and privileged; user applications call into it for sensitive work (open file, allocate memory, send packet). The file system adds names, folders, permissions, and metadata (created date, size). The shell is your interface: a GUI renders windows and touch targets; a CLI accepts text commands and scripts—critical for repeatability and remote admin.

Why CLI matters on exams: Scripting, remote repair, and repeatable fixes. The GUI hides steps; the CLI shows exact commands for logs and automation.

Application architecture (how software is delivered)

Where code executes changes updates, offline use, and security boundaries. A thick client installs binaries locally—fast UI, works offline (often), but patching is your responsibility. A web app ships logic from a server each visit; thin local footprint, but you need connectivity (or progressive offline tricks) and you trust the browser sandbox. A thin client / VDI / DaaS model streams pixels or remotes a session: the data often never lands on the endpoint, which helps loss/theft scenarios but demands a solid network.

  • Local / “thick” client — Installed on disk; runs mostly on local CPU/RAM (office suites, games).
  • Web app — Runs in the browser; UI in HTML/JS; data often on remote servers.
  • Thin client / streaming — Little local processing; displays remote desktop or hosted session (DaaS fits here).

Updates, licensing, and hygiene

Updates are not “annoying reboots”—they ship security patches for bugs attackers already study. Deferring them increases risk. Licensing tells you how many installs, whether VMs are allowed, and who you can call for support. OEM licenses ride with a machine; retail/volume licenses transfer differently. Open source is not automatically “no rules”—it has licenses too (GPL vs MIT differ in obligations). Pirated keys and cracked installers are a malware distribution channel and a compliance failure.

Web browser configuration (high-yield for Tech+)

The browser is the user’s most attacked daily app: it fetches untrusted code (JavaScript), renders it, and often stores credentials. Configuration is defense in depth: reduce attack surface (extensions), limit tracking (cookies, third parties), and keep secrets out of easy Autofill on shared PCs.

  • Private / InPrivate / Incognito — Reduces local history/cookies for that window; does not hide traffic from employer, ISP, or websites.
  • Extensions — Can read page content; remove unused ones; prefer official stores; review permissions.
  • Cache — Stores images/scripts for speed; clearing fixes stale pages and some login glitches.

  • Pop-up blockers — Stop malicious or annoying windows; may need exceptions for legitimate apps.
  • Bookmarks / favorites — Organize; avoid typosquatting URLs.

How programs run: interpreted vs compiled vs query

Source code is human-readable; processors execute numeric instructions. A compiler bridges that gap ahead of time: it analyzes the whole program (syntax, types), emits machine code or bytecode, and may optimize. You get faster startup at run time and errors caught before deployment—but you must rebuild when code changes.

An interpreter (or runtime) reads the program as it runs, which makes iteration fast and deployment simple (“just copy the script”). Hybrid systems (Java on the JVM, Python with caches) blend both worlds. Neither is “better”—they fit different domains.

SQL is declarative: you describe what set of rows you want; the database engine decides how to scan indexes and join tables. That is why SQL sits in its own column below—it talks to data, not directly to the CPU instruction stream like C.

CategoryIdeaExamples (illustrative)
CompiledSource translated to machine code before run; often catches errors at build timeC, C++, Rust, Go (AOT style)
Interpreted / hybridSource or bytecode executed by runtime; flexible, portable; may use JITPython, JavaScript, Java on JVM
Query languageDeclares what data you want from a databaseSQL (SELECT, INSERT, UPDATE, DELETE)

An IDE bundles editor, debugger, build tools, and sometimes visual designers—faster than Notepad for real projects.

Data types (primitive concepts)

Types tell the language how to store bits and which operations are legal. Adding two integers is arithmetic; adding two strings might mean concatenation—same plus sign, different meaning. Floats approximate real numbers and can surprise you with rounding (money apps often use integers of cents). Booleans drive every decision point.

  • String — Sequence of characters; "hello".
  • Integer — Whole numbers; -3, 0, 42.
  • Float / double — Numbers with fractional parts; scientific and currency math (watch rounding).
  • Boolean — true or false; drives conditions.

Structures: variables through objects

Small programs can live in one file; real systems organize logic. A variable is a named box; a constant is a box you promise not to repack—useful for configuration like tax rates. An array lines up many values by index—great for loops. A function hides a recipe: parameters in, return value out, reusable everywhere.

Object-oriented thinking models nouns in the problem domain: a “customer” has properties (name, balance) and methods (placeOrder, resetPassword). The class is the cookie cutter; an object is one cookie. Tech+ will not make you code a full class, but it will ask whether a snippet behaves like data+behavior.

  • Variable — Named storage whose value can change (depending on language rules).
  • Constant — Value fixed after assignment; prevents accidental overwrite.
  • Array / list — Ordered collection accessed by index; same type or mixed depending on language.
  • Function / method — Named block with parameters and return value; reduces duplication.
  • Object — Combines properties (data) and methods (behavior); foundation of OOP.

Logic: branching and looping

Programs are mostly sequence, selection, and repetition. Branching answers “under what condition do we take path B?” Loops answer “how many times?” or “until what becomes false?” Exam pseudocode may nest loops inside branches—trace carefully, writing down variable values on paper if needed.

Branching — if / else if / else, switch on discrete values, short-circuit evaluation. Loops — for (known iterations), while (condition-driven), foreach over collections. Watch off-by-one errors and infinite loops.

Pseudocode & flowcharts

Teams sketch before they code. Pseudocode is a structured English-ish outline—indentation shows nesting; it reads like code but ignores strict syntax, so focus on sequence, conditions, and loops. Flowcharts are graphical; the exam cares that you follow arrows through decisions without skipping a branch. Neither format runs on a CPU; both exist to communicate intent and catch logic errors cheaply.

Sample pseudocode — print even numbers 2–10:
FOR n FROM 2 TO 10 STEP 2
  PRINT n
END FOR

Markup vs programming vs assembly

HTML and XML are markup languages—they describe structure (headings, paragraphs, tags), not full algorithms. Browsers render HTML; parsers validate XML schemas. Do not call HTML a “compiled programming language.”

Scripting languages (JavaScript in browsers, Python for automation, PHP on servers) are often interpreted or JIT-compiled—flexible for glue code and web.

Assembly is a low-level mnemonic view of machine instructions—tied to a CPU architecture. Compilers may emit assembly as an intermediate step; humans use it for drivers, bootloaders, and extreme optimization, not daily app dev on Tech+.

ASCII, Unicode, and “why emoji work”

ASCII maps English letters, digits, and punctuation to 7-bit codes (128 characters)—fine for legacy English text. Unicode is a universal character set that includes accented letters, scripts worldwide, symbols, and emoji. UTF-8 encodes Unicode in bytes and stays compact for English while scaling to international text. Exam angle: Unicode is a superset of ASCII for practical purposes.

Second trace: nested branch + loop

Pseudocode — print “OK” for scores ≥70, else “Retry,” but skip negatives:
FOR EACH score IN list
  IF score < 0 THEN SKIP
  ELSE IF score >= 70 THEN PRINT "OK"
  ELSE PRINT "Retry"
END FOR
Trace with [85, -5, 60]: prints OK, (nothing for -5), Retry.

Version control (e.g., Git) tracks changes to source over time—rollback, branches, collaboration. You may see it as “where developers store code history,” not something you configure on Tech+.

Why data is valuable

Organizations do not collect data for sport—they expect better decisions, automation, or revenue. A retailer studies baskets to stock shelves; a school studies grades to target tutoring; a factory studies sensor streams to predict maintenance. When data is wrong, incomplete, or biased, models and dashboards lie confidently. That is garbage in, garbage out. Regulations exist because personal data can harm people if leaked or misused: GDPR in the EU, sector rules like HIPAA-style protections for health data, and state privacy laws in the US.

Analytics, big data, and data-driven decisions

Analytics is the everyday work of SQL queries, spreadsheets, and BI dashboards—datasets that still fit mostly in one place. Big data is the regime where volume, speed, or variety forces distributed systems: think clickstreams, genomics, or city-wide IoT. You do not need to engineer Hadoop for Tech+; you need to recognize why classic relational databases strain at that scale. Data-driven culture means leadership asks for evidence—but evidence is only as good as governance and definitions.

  • Analytics — Dashboards, reports, statistics on datasets you can reasonably move into one tool.
  • Big data — Often described with volume, velocity, variety; needs distributed storage and processing (Hadoop/Spark-style ecosystem at a high level).
  • Data-driven decision — Leadership chooses actions supported by metrics, not only gut feel.

Database fundamentals

A flat file (one giant spreadsheet or CSV) works until two people edit at once, until you need enforced rules (“order total must match line items”), or until you need fast lookups on millions of rows. A database adds a DBMS engine that handles concurrency, crash recovery, indexing, and permissions.

Relational model — Data in tables (relations). Each row is a record; each column is a field with a type. A primary key uniquely identifies a row. A foreign key references a primary key in another table, enforcing relationships (customer → orders).

Non-relational (NoSQL) — Document stores (JSON documents), key-value caches, wide-column, graph DBs—chosen when schema flexibility or massive scale matters.

Think of schema as a contract: column names, types, and rules. Changing schema without planning breaks apps. CRUD maps to SQL verbs you should recognize: create rows (INSERT), read (SELECT), update (UPDATE), delete (DELETE). Structure changes use CREATE/ALTER/DROP on tables.

SQL pattern (recognition only): SELECT name, email FROM customers WHERE state = 'TX'; — retrieves columns from a table with a filter.

Common app actions acronym CRUD: Create, Read, Update, Delete.

Backups: scope and location

Backups are an insurance policy—useless if you never test a restore. Ransomware made offline or immutable copies famous: if your only backup is a mapped drive letter, malware encrypts it too. File-level backup protects Sally’s Documents; image backup captures OS + apps so you can rebuild a machine after disk death without two days of reinstalling.

Local copies are fast to restore; off-site/cloud copies survive building loss. Rotate media, encrypt sensitive archives, and document who is allowed to run restores (least privilege applies to backup consoles too).

  • File / folder backup — User documents, photos, databases export—fast restore of data but not full OS state.
  • System / image backup — Snapshot of OS, apps, drivers—bare-metal or full machine restore.
  • Local — External drive, NAS on LAN; quick restores.
  • Cloud / off-site — Survives fire/theft; watch encryption and retention policies.
3-2-1 rule (classic): At least three copies, on two different media, with one off-site. Test a restore before you need it.

Full vs incremental vs differential backups

TypeWhat is copiedRestore complexity

Cloud sync (OneDrive/Google Drive) is not a full image backup—it may not recover OS state, and ransomware can encrypt synced copies if versioning is weak.

Normalization (one-sentence version)

Normalization splits redundant data into related tables so you update a fact in one place (e.g., customer address lives in customers, not repeated on every order row). Over-normalization can mean many joins for simple reports—denormalized star schemas appear in analytics warehouses for speed.

ETL, warehouses, and BI (vocabulary)

ETL (Extract, Transform, Load) pulls data from sources, cleans/conforms it, and loads it into a warehouse. A data warehouse is a central store optimized for reporting across systems. Business intelligence (BI) tools build dashboards from that data. Tech+ may contrast “operational OLTP database” (many small fast transactions) with “analytical” workloads (big scans, aggregates).

More SQL patterns (recognition)

Filter + sort: SELECT product, price FROM inventory WHERE price < 100 ORDER BY price DESC;
Join (combine tables): SELECT orders.id, customers.name FROM orders JOIN customers ON orders.customer_id = customers.id; — links rows where foreign key matches primary key.
Insert / update / delete: INSERT INTO adds rows; UPDATE changes existing rows (always use a WHERE or you update everything); DELETE removes rows (same WHERE warning).

A database dump is often a logical export (SQL statements or portable format)—restoring may require an empty database first. A bare-metal or image backup captures the whole machine state.

Security on this exam is practical end-user and small-office level: recognize threats, apply sane defaults, and explain why a control exists—not exploit development or enterprise SOC playbooks.

CIA triad and access control

Security goals compete with convenience and cost. The CIA triad is the shorthand examiners use to sort controls. Confidentiality asks: “Could someone who should not see this still see it?” Encryption, file permissions, and training reduce leaks. Integrity asks: “Was this altered in transit or on disk without us knowing?” Signed updates, hashes, and access logs support integrity. Availability asks: “Can legitimate users reach the service when they need it?” Redundant power, backups, and DDoS defenses protect availability—an attacker who cannot steal data may still try to knock you offline.

  • Confidentiality — Only authorized users/systems read secrets (encryption, permissions, clean desk).
  • Integrity — Data and configs are accurate and unmodified (hashing, patching tampering, change control).
  • Availability — Services stay up (redundancy, backups, DDoS mitigation, UPS).

Authentication answers “who are you?” Authorization answers “what may you do?” A receptionist might authenticate to the network but not be authorized to open payroll. Least privilege limits blast radius when credentials leak. MFA adds a second factor—something you have (token) or are (biometric)—so stolen passwords are not enough.

Threats: phishing, social engineering, malware

Many breaches start with a person, not a missing patch. Phishing clones trusted brands, invents urgency (“invoice overdue”), and links to fake login pages. Spear phishing uses your name and context. Vishing and smishing move the same tricks to phone and SMS. The defense is skepticism + verification: call the company on a known number, don’t use the one in the suspicious message.

Social engineering exploits helpful instincts: tailgating through a door, USB drives labeled “Payroll” left in a parking lot, fake IT calls. Policy + awareness + physical controls beat pure technology here.

Malware typeHigh-level behavior

RansomwareEncrypts data; demands payment—backups are the recovery story

Story thread: User opens a “shipping notice” attachment → macro runs → trojan drops spyware → credentials exfiltrated → attacker uses ransomware weeks later. Good backups and least-privilege accounts limit damage; layered defenses catch different stages.

Device hardening

Hardening is subtraction: remove what you do not need, patch what you keep, and monitor the rest. Unused services are latent vulnerabilities. AutoPlay launching USB content is an easy policy win. Host firewalls block unexpected inbound; anti-malware adds signature and behavior checks. Full-disk encryption protects confidentiality if a laptop is stolen—not availability if the thief formats the drive, but your data does not walk away readable.

  • Patching — OS, firmware, apps; prioritize critical CVEs.
  • Anti-malware / EDR — Real-time scanning, behavioral detection.
  • Firewall — Host firewall blocks unexpected inbound; network firewall segments LAN from WAN.
  • Disable unused services / ports — Shrinks attack surface.
  • Encryption at rest — BitLocker/FileVault/device encryption for lost hardware.

Safe software practices

Honor licenses and EULA (seat count, virtualization rights). Download only from vendor or first-party stores; verify checksums when provided. Piracy risks fines, malware, and no support.

Passwords and encryption in transit

Humans reuse passwords because memory is limited—attackers exploit that with credential stuffing (try leaked email/password pairs everywhere). Unique passwords per site contain the blast radius. A password manager generates entropy-heavy secrets and resists phishing by filling only matching domains.

Data at rest sits on disk or flash; steal the device, you have the bits—unless encryption makes them ciphertext. Data in transit crosses Wi-Fi, ISP links, and data centers—any hop can be observed on hostile networks. HTTPS wraps HTTP in TLS: the browser validates the server certificate chain before exchanging keys. A VPN adds another encrypted tunnel—valuable on untrusted Wi-Fi or for remote access to a corporate LAN— but you still must trust the VPN operator and keep endpoints patched.

SSL/TLS note: SSL is legacy naming; modern sites use TLS. Exams may still say SSL colloquially.

Wireless security checklist

Home and small-office routers are full computers with default passwords printed on stickers—bots scan for them constantly. WPA2/WPA3 with a strong passphrase beats “open” Wi-Fi for confidentiality of wireless frames. Guest networks isolate visitors from file shares and printers. Firmware updates close remote-management bugs; disabling unused features (WPS where flawed, cloud admin if not needed) shrinks attack surface.

  1. Change default router admin password and disable remote admin unless required.
  2. Set a non-default SSID (hiding SSID is not real security).
  3. Use WPA3 (or WPA2-AES); never WEP or open networks for sensitive work.
  4. Enable guest network for visitors; isolate from LAN.
  5. Update router firmware; disable WPS if it is a known weak point on that model.

Network attacks you can name

DoS / DDoS flood a service with traffic or requests so legitimate users cannot get through—attacks availability. Mitigation includes ISP scrubbing, rate limits, redundant capacity, and planning.

MITM (man-in-the-middle) intercepts or alters traffic between two parties—public open Wi-Fi plus fake captive portals is a classic setup. HTTPS and VPN reduce readability of payloads on the wire, but users must still verify certificates and avoid bad proxies.

Spoofing forges sender identity (email “from” address, caller ID, fake websites). Replay resends captured legitimate packets—protocols use nonces/timestamps to fight this at higher levels.

Email and messaging hygiene

SMTP sends; POP3 and IMAP receive—prefer IMAP for multi-device sync. SPAM is unsolicited bulk mail; filters score content and reputation. Phishing often arrives as email or SMS with malicious links/attachments. SPF/DKIM/DMARC are sender-authentication technologies that reduce forged mail—know them as “help prove mail really came from that domain.”

Never enable macros on unexpected Office files. Block .exe attachments at the mail gateway when policy allows; train users to verify invoices and HR requests through known channels.

Certificates, hashing, and encryption vocabulary

A digital certificate binds a public key to an identity (website, organization) signed by a Certificate Authority (CA). Browsers trust built-in root CAs and validate chains during HTTPS. A hash (e.g., SHA-256) produces a fixed-length fingerprint of data—tiny changes produce totally different hashes, useful for integrity checks of downloads.

Symmetric encryption uses one shared secret (fast, used for bulk data). Asymmetric uses key pairs (public encrypt / private decrypt, or signing)—slower, used to bootstrap symmetric keys (as in TLS handshakes). You will not implement these on Tech+; you will classify scenarios.

Physical security and insider risk

Tailgating follows an authorized person through a door—defeat with badges, mantraps, and culture (“please badge in”). Clean desk hides sensitive papers. USB drops (malicious flash drives) exploit curiosity—disable AutoRun and use policy. Screen privacy filters reduce off-angle viewing; Win+L / lock when stepping away.

Insider threats include disgruntled employees and compromised accounts—logging, least privilege, and separation of duties limit damage.

Exam Essentials — security threats & best practices

Threats, CIA, privacy (Ch.10)

  • Data is an asset—used to improve products, target customers, or sold; critical data needs strongest controls.
  • Big data = large, often unstructured datasets analyzed for patterns.
  • Confidentiality attacks: snooping, eavesdropping, wiretapping, social engineering, dumpster diving.
  • Integrity attacks: on-path, replay, impersonation, unauthorized alteration.
  • Availability attacks: DoS, outages, power loss, hardware failure/destruction.
  • PII privacy; GDPR (EU); US Privacy Act + state laws; cookie consent before storing cookies.
  • Social media: public vs friends-only expectations.
  • Authentication factors: password, PIN, OTP, software/hardware tokens, biometrics, location, security questions.
  • SSO one credential set across systems.
  • Permissions bundle privileges (read vs edit).
  • Least privilege minimum access needed.
  • Accounting via logs, location tracking, browser history.
  • Nonrepudiation: video, biometrics, signatures, receipts.
  • Malware family: viruses, worms, trojans, adware, spyware, ransomware, rootkits, backdoors, spam, password attacks.

Hardening, passwords, encryption, browsing (Ch.11)

  • Device hardening: anti-malware + software firewall; non-default passwords; safe browsing; patch; trusted download sources; remove/disable unused software & services; disable AutoPlay.
  • Physical: laptop cable lock; Ctrl+Alt+Del / lock when leaving seat.
  • Encryption at rest scrambles data so even local admins can’t read others’ encrypted files (user-specific).
  • In transit: corporate email often encrypted; HTTPS; VPN for remote workers—client makes remote LAN look local.
  • Passwords: change defaults; complexity; confidential; rotate; no reuse across sites or recycled old passwords.
  • Browser: latest version; manage extensions; disable risky Autofill; security settings; cookies/history hygiene.
  • Secure site: URL starts with https://.
  • Policies: acceptable use, confidentiality handling, security awareness training.

How to learn acronyms as language, not flashcards alone

Three-letter acronyms stick when they sit inside a sentence you could explain to a friend. For each term, ask: What layer is it? (hardware, OS, network, app). What problem does it solve? What breaks if it is misconfigured? Example: DHCP is not “a magic setting”—it is how a host learns IP, mask, gateway, and DNS without manual typing. DNS is not “the Internet”—it is name resolution; break DNS and IP-only tests still work.
